# Sandboxed Code Execution Service

We're building a sandboxed code execution service — the kind of system that powers platforms like LeetCode, Codeforces, and Judge0 under the hood. The core problem it solves is trust: when a platform lets strangers submit arbitrary code and runs it on shared server infrastructure, that code could accidentally or maliciously hang the server with an infinite loop, exhaust memory or spawn endless processes, read files it shouldn't (other users' data, secrets, credentials), or reach out over the network to attack other systems. The project's job is to take untrusted user-submitted code and execute it safely by stripping away, layer by layer, the default capabilities any process would normally have — enforcing time limits, memory/process limits, filesystem isolation, and (as a stretch) network isolation and syscall restrictions — so the code can compute its answer and nothing else. Alongside the sandboxing itself, it's also a real backend engineering exercise: an API that accepts submissions, a worker pool that executes them concurrently and safely, and a result-reporting layer, all built to survive genuinely adversarial input.

## Threat Model Log

Phase 1 writes each submission to its own temp directory, but this is filesystem convenience, not isolation — the executed process can still read/write anywhere the server process can, spawn other processes, and reach the network.

Phase 2 adds a hard time limit via `context.WithTimeout` + `exec.CommandContext`, so a hung submission is killed and the server stays responsive. However, this only signals the direct child process — a submission that forks children (e.g. a fork bomb) can leave orphaned processes running after the parent is killed. Process-group-based cleanup is deferred to Phase 3.

Phase 3 closes the process-group and resource-exhaustion gaps left by Phase 2. Every submission now runs inside its own cgroup v2 (`pids.max`, `memory.max`), scoped per-request rather than shared across the whole OS user. An initial `RLIMIT_NPROC` (`ulimit -u`) approach was abandoned after it took down the entire dev VM: `RLIMIT_NPROC` is scoped by UID system-wide, not per-submission, so once a fork bomb pushed the shared user's process count to the cap, sshd itself couldn't fork a handler for new connections. `RLIMIT_AS` (`ulimit -v`) was likewise replaced by cgroup `memory.max`, which triggers an uncatchable, per-cgroup OOM kill instead of a catchable Python `MemoryError`. The submission's process now also runs in its own process group (`Setpgid`), and both the timeout and a new 1MB stdout/stderr cap kill the whole group via `syscall.Kill(-pgid, SIGKILL)` instead of just the direct child, closing the orphaned-children gap. Filesystem and network isolation remain out of scope — Phase 4.

Phase 4 closes the filesystem and process-visibility gaps flagged at the end of Phase 3. Each submission now runs inside a `chroot`'d jail — a minimal root filesystem containing just the Python interpreter, its shared libraries, and the standard library — combined with a new mount namespace (`CLONE_NEWNS`) and PID namespace (`CLONE_NEWPID`). The process can no longer read anything outside its own jail (`/etc/passwd`, the server's own source, other submissions' temp directories all return "not found"), and it can no longer see any process on the host: it shows up as PID 1 in its own namespace, with a freshly mounted `/proc` scoped to just itself rather than inherited from the host. A classic `chroot`-only escape (repeatedly `mkdir` + `chdir`, then `chroot('.')` from the deepest directory to climb past the jail root) was tested directly against this setup: the syscalls succeed, since the sandboxed process still runs as root, but the mount namespace means there is no host filesystem tree reachable to climb into — the escape lands in an empty, self-created directory rather than the real root. Syscall restriction (seccomp) remains out of scope, as noted in the intro above — see Phase 6 for network isolation.

Phase 5 turns single-submission handling into a bounded worker pool so multiple submissions execute concurrently instead of serializing behind one HTTP handler. A fixed pool of goroutines (`runtime.NumCPU()`, floored at 4 to stay useful on this 2-vCPU dev VM, since each worker spends nearly all its time blocked on `cmd.Wait()` rather than burning CPU) pulls jobs off a 100-slot buffered channel; `/submit` now builds a `Job`, pushes it with a non-blocking send, and returns `503` immediately if the queue is full rather than let requests pile up unbounded — the buffer absorbs bursts, the 503 is the honest answer once it's genuinely saturated. A separate 30s queue-wait timeout (distinct from the existing 5s execution timeout) protects against a request waiting forever if it never reaches a worker, returning `queue_timeout` instead of hanging; a job's per-job result channel is deliberately buffered (capacity 1) so a worker that finishes after its caller has already timed out can still deliver its result without blocking forever, which would otherwise leak one worker per timeout. Per-job resource paths (temp dirs, cgroups) were audited rather than changed: `os.MkdirTemp`'s uniqueness guarantee already makes cgroup paths collision-free under concurrency, since each is a pure function of an already-unique temp dir. `go run -race` confirmed no data races — the only state shared across worker goroutines is the job channel itself (self-synchronizing) and the stdlib `log.Logger` (documented safe for concurrent use), so no locks were needed anywhere. Verified with 8 concurrent 2-second submissions completing in ~4.5s wall-clock against 4 workers (vs. ~16s serial), and with the full Phase 2-4 evil test suite (fork bomb, memory bomb, infinite loop, output flood, chroot escape, host file read, process visibility) fired concurrently alongside normal submissions — every evil payload still hit its expected contained outcome, and normal jobs were neither delayed nor corrupted by running alongside them. One known gap: a `queue_timeout` response does not cancel the underlying job — it still runs to completion in its worker, consuming capacity for a client that's already given up. True cancellation would need a `context`/cancellation signal threaded through `Job` into `executeJob`, deferred as a future improvement. Syscall restriction remains out of scope, as noted above — see Phase 6 for network isolation.


Phase 6 closes the last named gap from Phase 4: network access. Each submission's process now also gets a new network namespace (`CLONE_NEWNET`), added alongside the existing mount and PID namespaces in the same `Cloneflags` value. A fresh network namespace starts with zero interfaces — not even loopback is brought up — so the flag alone provides full isolation with no veth pairs, bridges, or NAT rules required, unlike a design that wanted to grant *limited* network access rather than none. Loopback was deliberately left down: submissions are algorithmic problem solutions with no legitimate reason to open a socket to themselves. Verified with three new evil tests: an outbound HTTP request (`http_request.py`), a raw socket connect to both a public IP and 127.0.0.1 (`raw_socket.py`), and a DNS lookup (`dns_lookup.py`) — all three fail as expected (`Network is unreachable` for the raw socket attempts, `Temporary failure in name resolution` for the DNS-dependent ones), and the full Phase 1-5 evil test suite plus normal submissions still pass unaffected running alongside them. Syscall restriction (seccomp) remains the one item still out of scope, flagged since Phase 4.

Phase 7 closes the last item flagged since Phase 4: kernel-level syscall filtering via seccomp-bpf, so a submission is blocked from making a dangerous syscall even if every earlier layer (chroot, namespaces, cgroups) somehow had a gap. **Status: implemented and verified on the Multipass VM.** The code was written on a macOS dev machine — `libseccomp-golang` is a cgo binding over the C `libseccomp` library and only compiles on Linux with `libseccomp-dev` installed, so nothing here could be built or tested until it ran on real Linux. Every gap below was found and fixed through that VM verification loop, not guessed in advance; the milestone this phase set out to hit — legitimate submissions succeed under the filter, a ptrace escape is blocked at the kernel level, and every prior phase's evil tests still pass with the filter active — is confirmed. What follows is the design, the whitelist, and the full trail of what broke and why, left in rather than cleaned up, since the debugging trail is as informative as the end state.

Design: applying a seccomp filter to *just* the untrusted process, without restricting the trusted setup code that mounts `/proc` and chroots, isn't straightforward from Go — `os/exec`'s `SysProcAttr` has first-class support for `Chroot` and `Cloneflags` but nothing for seccomp, and manually forking inside a multithreaded Go runtime to install a filter pre-exec is unsafe (only the calling thread survives a raw `fork()`, leaving every other goroutine's locks in an inconsistent state). The fix is a self-re-exec: `worker.go` no longer runs `sh -c "mount ... && exec python3 ..."` directly. Instead it launches the *server binary itself* with `Cloneflags` (new mount/PID/net namespaces, same as Phase 4/6) but deliberately without `Chroot`, as `<server-binary> __sandbox_init__ <jailPath> <solutionPath>`. `main.go` detects that argv0 and hands off to `runSandboxInit` (`seccomp.go`), which — now running as a freshly exec'd, single-threaded process, not a fork of the multithreaded parent — does, in order: mount `/proc` into the jail, `chroot` into it, build and load the seccomp filter, then `syscall.Exec` into `python3`. Because a loaded seccomp filter persists across `execve`, python3 inherits it for its entire lifetime, while the mount/chroot/filter-building steps that precede it were never themselves filtered.

The filter is a whitelist. Default action for anything not listed was `SCMP_ACT_ERRNO` (`EPERM`) throughout development — a killed process gives no diagnostic about which syscall it needed, whereas `EPERM` surfaces as a normal, readable error — and only flipped to `SCMP_ACT_KILL` (see below) once the whitelist had been fully verified against it. The whitelist in `seccomp.go` is grouped by category: memory management (`mmap`, `munmap`, `mprotect`, `brk`, `mremap`, `madvise`), process/thread bookkeeping (`exit`, `exit_group`, `gettid`, `getpid`, `set_tid_address`, `futex`, `sched_yield`, `prlimit64`, ...), signals (`rt_sigaction`, `rt_sigprocmask`, `rt_sigreturn`, `sigaltstack`, `tgkill`), file I/O scoped to the jail (`read`, `write`, `openat`, `close`, `fstat`, `lseek`, `fcntl`, `getdents64`, ...), time (`clock_gettime`, `nanosleep`, ...), and interpreter-startup misc (`uname`, `getrandom`, `prctl`). It started as a draft assembled from well-documented CPython syscall usage on the macOS dev machine, then was corrected against real `strace` output and real submissions on the VM — see the correction log below for exactly what that process found and fixed.

Deliberately excluded, and why: `clone`/`clone3`/`fork`/`vfork` — submissions are single-threaded algorithmic solutions with no legitimate reason to spawn a process or thread, so excluding these makes the Phase 3 fork-bomb defense redundant at the kernel level too, not just via `pids.max`. `execve`/`execveat` were initially excluded on the same reasoning, but that broke every submission: the filter is loaded by `runSandboxInit` itself, immediately before it calls `execve` to become `python3`, and a loaded filter applies to that call too, not just to what comes after it — a bug only visible once this ran on real Linux (see the "on-VM correction" note below). `execve` is now whitelisted; the residual risk of user code calling `os.execve()` is mitigated by the jail's contents rather than the filter — there is nothing but the `python3` binary and its own libraries inside the chroot for a stray exec to target. `ptrace` — the classic sandbox-escape/debugging vector this phase's evil test (`tests/evil/ptrace_attempt.py`) targets directly. `mount`/`umount2` — filesystem manipulation; the submission's own process never needs to mount anything (that already happens in the trusted `sandbox-init` step before the filter loads). `chroot` — the submission process has no legitimate reason to call it itself; confirmed on the VM (see correction log below), where `tests/evil/escape_attempt.py`'s outcome changed from Phase 4's "chroot succeeds but lands in an empty, self-created directory" to an outright `PermissionError` at the syscall level — a strictly stronger result, defense in depth on top of the existing mount-namespace containment. `socket`/`connect`/`bind`/accept-family calls — redundant with Phase 6's empty network namespace, kept out of the syscall filter too as defense in depth. `reboot`, `kexec_load` — no legitimate use, obviously. One additional hardening choice: the filter is native-architecture-only (no 32-bit/x32 compatibility syscall table is added), since mixing syscall ABIs into one filter is itself a known seccomp bypass vector.

On-VM correction log (2026-07-20, Multipass VM): `strace -f -c python3 solution.py` across print/loop/math/file-I/O submissions showed the guessed whitelist was missing `getuid`/`geteuid`/`getgid`/`getegid` (CPython's startup checks these unconditionally) — added. Then the first real end-to-end `/submit` call failed every time with `sandbox-init: exec python3: operation not permitted`: the seccomp filter, loaded inside `runSandboxInit` right before it calls `syscall.Exec` to become `python3`, was itself blocking that `execve` call, since a loaded filter applies to every syscall the calling thread makes afterward — including the one used to hand off to the sandboxed binary. Fixed by whitelisting `execve` (see the "deliberately excluded" paragraph above for the updated reasoning). This is exactly the kind of gap the phase brief warned about ("a common failure mode: your filter accidentally blocks a syscall your own jail/cgroup setup code needs") — it just turned out to be the filter blocking *itself*, not the jail/cgroup code.

A first full evil-suite run (`evil_load_test.sh`) came back looking inconsistent — `raw_socket.py` and `fork_bomb.py` each appeared to fail two different ways across runs. That turned out to be a test-harness bug, not a sandbox bug: the script backgrounded every submission's `curl | python3 -m json.tool` writing straight to the shared terminal, and JSON blobs larger than one pipe buffer (4KB) interleaved mid-write across concurrent jobs, producing spliced/corrupted output that looked like nondeterminism. Fixed by having each job write to its own temp file and printing them sequentially after `wait`. Re-run with the fix, results were clean and consistent: every prior evil test (`fork_bomb`, `memory_bomb`, `infinite_loop`, `output_flood`, `read_host_file`, `list_processes`, `http_request`, `raw_socket`, `dns_lookup`) still lands on its expected contained outcome, `escape_attempt.py`'s outcome changed as described above (chroot now blocked outright), and four concurrent normal submissions all completed successfully. The one test that still doesn't verify anything about seccomp: `ptrace_attempt.py` fails at `import ctypes` with `ImportError: libffi.so.8: cannot open shared object file` — the jail's Python build is missing `libffi`, a `ctypes`/`_ctypes` runtime dependency that `ldd python3` doesn't reveal since it's loaded via `dlopen` at import time, not linked directly. This is a jail-template packaging gap, unrelated to the seccomp filter itself, and still needs fixing before the ptrace evil test can actually run.

Fixed on the VM by copying `libffi.so.8` (and the real file it symlinks to, `libffi.so.8.2.0` — `cp -a` preserves a symlink as a symlink, it doesn't also pull in its target) from `/usr/lib/aarch64-linux-gnu/` on the host into the matching `/opt/jail-template/usr/lib/aarch64-linux-gnu/` directory. **Note for future jail-template rebuilds**: `/opt/jail-template` lives on the VM, outside this repo's git history, so this dependency isn't captured anywhere version-controlled — if the jail template is ever regenerated from scratch, `libffi` needs to be added back manually, since it's a `ctypes`/`_ctypes` runtime dependency that a plain `ldd python3` won't reveal. With it in place, `ptrace_attempt.py` finally exercises the real thing: `ptrace(PTRACE_TRACEME)` returns `errno=1` (EPERM), confirming seccomp blocks the classic ptrace-based sandbox-escape/debugging vector at the kernel level — the phase's core milestone.

The switch to `SCMP_ACT_KILL` immediately caught a gap that `SCMP_ACT_ERRNO` had been silently hiding: every single submission, including plain `print(2+2)`, started dying with empty stdout/stderr — a strong signal something in interpreter startup itself was failing, not something evil-test-specific. `sudo dmesg | grep -i seccomp` confirmed it directly: `comm="python3" ... syscall=38` (on arm64's syscall table, `renameat`). CPython writes its `__pycache__` bytecode cache atomically on every module import (temp file + rename); under `SCMP_ACT_ERRNO` that write silently failed and Python just skipped caching, invisible to every test run so far, but under `SCMP_ACT_KILL` the same missing syscall killed the process outright. This is exactly why `SCMP_ACT_ERRNO` was the right default *during* whitelist discovery, as the phase brief recommended — the failure mode it produces (a caught, often-ignorable error) can mask real gaps that only surface once you actually flip to kill mode. `renameat`/`renameat2` are now whitelisted, and the full evil suite plus concurrent normal submissions were re-verified clean under `SCMP_ACT_KILL` on the VM — the final, current state: `fork_bomb`, `raw_socket`, `http_request`, `dns_lookup`, `escape_attempt`, and `ptrace_attempt` are all killed by `SIGSYS` at the syscall boundary (clone, socket/connect, chroot, ptrace — none whitelisted); `infinite_loop`, `memory_bomb`, `output_flood`, `list_processes`, and `read_host_file` are unaffected, still caught by their own Phase 2/3/4 mechanisms exactly as before; and four concurrent normal submissions all complete with `"status":"success"`.

What's still open, honestly: the whitelist has only been exercised against a handful of code paths (print, a tight loop, float/string ops, in-jail file read+write, module import/bytecode caching). It's plausible other legitimate code paths — different stdlib modules, different I/O patterns — need syscalls not yet on the list, and given the `renameat` experience, any such gap could be silently masked if `SCMP_ACT_ERRNO` is ever reverted to for debugging without a full re-verification at `SCMP_ACT_KILL` afterward. Broader syscall coverage (more varied strace runs, more stdlib modules exercised) would be the natural next increment beyond what this phase attempted.
